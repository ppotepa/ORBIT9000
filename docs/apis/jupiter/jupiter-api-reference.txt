    Base URLs and API Access: Jupiter provides Lite (free tier) API endpoints under the base https://lite-api.jup.ag (no API key required) and Pro endpoints under https://api.jup.ag (API key required via x-api-key header)[1]. Unless otherwise noted, methods are RESTful HTTP calls. Jupiter currently does not expose GraphQL or WebSocket endpoints; all APIs are HTTP requests. Each section below is organized by service (Ultra, Swap, Lend, Trigger, Recurring, Token, Price). For Pro usage, ensure you include a valid API key in the request headers.

    
    Ultra API

    The Ultra API is Jupiter’s next-generation swap API, providing an all-in-one solution for trading on Solana with built-in slippage protection, transaction broadcasting, MEV protection, and integrator fee support[2][3]. Ultra endpoints do not require a Pro API key (dynamic rate limiting applies)[4]. Base URL: https://lite-api.jup.ag/ultra/v1.

    
    GET /ultra/v1/order

    Description: Returns a base64-encoded unsigned swap transaction for a given token swap request[5]. The client must later sign this transaction and execute it (via the execute endpoint). This “order” response also contains a requestId used to correlate with execution[6].

    Authorization: No API key needed (Ultra uses dynamic rate limits).

    Required Query Parameters: (pass as URL query string)
- inputMint (string, required): The mint address of the input token[7].
- outputMint (string, required): The mint address of the desired output token[7].
- amount (string, required): Amount of the input token to swap, specified in the token’s smallest units (e.g. lamports for SOL or atomic units for SPL tokens)[8].
- taker (string, optional): The user’s Solana wallet address. If provided, the transaction will be prepared to debit this address; if omitted, an order is still returned but with no transaction field in the response[9].
- referralAccount (string, optional): A referral account public key for integrator fees (must be pre-created via Jupiter’s referral program)[10].
- referralFee (integer, optional): Referral fee in basis points (bps) to charge on this swap (50–255 bps allowed)[11] (Jupiter takes 20% of this fee)[12]. If provided, ensure a corresponding referral token account is set up for the output token mint (otherwise the swap will proceed without applying the fee)[13].

    Example Request: (fetching a swap transaction for 1 SOL to USDC)

    const orderResponse = await (
  await fetch('https://lite-api.jup.ag/ultra/v1/order?inputMint=So11111111111111111111111111111111111111112&outputMint=EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v&amount=100000000&taker=YourWalletAddress')
).json();
console.log(JSON.stringify(orderResponse, null, 2));
```[14]

**Response Structure:** On success (`200 OK`), returns a JSON object with key fields:  
- **`transaction`** (string): The unsigned swap transaction in base64 format, to be signed by the caller’s wallet[15].  
- **`requestId`** (string): The unique identifier for this order request, used when executing the swap[6].  
- Other metadata fields such as **`swapType`**, **`router`**, **`inAmount`**, **`outAmount`**, **`slippageBps`**, **`priceImpactPct`**, and a detailed **`routePlan`** describing how the trade will be split across liquidity sources[16][17].

### POST `/ultra/v1/execute`

**Description:** Executes a **signed swap transaction** previously obtained from `/order`. This endpoint broadcasts the signed transaction to the Solana network and returns its execution status[21][22]. Using Jupiter’s execute endpoint is optional – you may also send the transaction to the network yourself (see **Response**). Jupiter’s service handles transaction broadcast through multiple RPCs, monitors for completion, and applies strategies for fast, reliable confirmation[21].

**Authorization:** No API key needed (Ultra dynamic rate limit). Endpoint: `https://lite-api.jup.ag/ultra/v1/execute`[23].

**Request Body (JSON):**  
- **`signedTransaction`** (string, required): The base64-encoded signed transaction obtained from a prior `/order` response (after signing it with the user’s private key)?11†L125-L133?.  
- **`requestId`** (string, required): The `requestId` from the `/order` response that corresponds to this transaction?11†L125-L133?.

**Example Request:**  

```js
const executeResponse = await (
  await fetch('https://lite-api.jup.ag/ultra/v1/execute', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      signedTransaction: signedTxBase64,
      requestId: orderResponse.requestId
    })
  })
).json();
```?11†L135-L144?

- **`status`** (string): `"Success"` or `"Failed"`?11†L163-L171??11†L182-L189?.

- If successful: **`slot`** (string or number): the slot number at which the transaction was confirmed; **`inputAmountResult`**, **`outputAmountResult`**: actual amounts swapped (as strings)?11†L165-L173?; **`swapEvents`**: array of objects detailing each swap leg (with input/output mints and amounts)?11†L169-L177?.

**URL Path Parameter:**  
- **`<publicKey>`** (string, required): The base58 wallet address whose token balances to retrieve?54†L62-L69?.

### GET `/ultra/v1/routers`

**Description:** Returns the list of **available routing engines (routers)** that Ultra API can use?2†L55-L60?. (Currently, Jupiter’s Ultra uses the **Juno** liquidity engine, including Metis and RFQ routing, as well as potentially third-party sources?6†L75-L83?.) This endpoint’s response enumerates the router names or identifiers available.

**Authorization:** No key needed.

**Response:** An array or object listing router names. (For example, it may include entries like `"Juno"` (aggregated engine), `"JupiterZ"` (RFQ engine), etc.) *This endpoint’s exact output format is not explicitly documented, but it is intended for informational use.*?2†L55-L60?

## Swap API (v6)

The **Swap API (v6)** is Jupiter’s earlier, lower-level swap API that directly interfaces with the Jupiter Metis v1 routing engine?14†L59-L67?. It allows fine-grained control over swap transactions, at the cost of more complexity compared to Ultra API. Swap API is recommended if you need custom instruction composition, CPI integration, or manual control of slippage, routing, etc?14†L79-L88?. Otherwise, prefer Ultra API for simplicity?14†L75-L83?. Base endpoints: `https://lite-api.jup.ag/swap/v1/*` (Lite) or `https://api.jup.ag/swap/v1/*` (Pro, with API key).

### GET `/swap/v1/quote`

**Description:** Fetches a **swap quote** for a given input/output token pair and amount. The quote includes the optimal route plan, projected output amount, price impact, slippage tolerance, and other details?15†L73-L81??15†L123-L131?, but does *not* create or execute a transaction. This is typically the first step: get a quote, then use it to build and execute a swap transaction.

**Authorization:** Lite (no key) or Pro (with key).  
**Endpoint:** `GET https://lite-api.jup.ag/swap/v1/quote` (use `api.jup.ag` for Pro)?15†L53-L60?.

**Required Query Parameters:**  
- **`inputMint`** (string): Mint address of the input token (source)?15†L97-L100?.  
- **`outputMint`** (string): Mint address of the output token (destination)?15†L99-L102?.  
- **`amount`** (string): The amount of input token (in base units, e.g. atomic units) to swap?15†L100-L107?. This should be the integer amount *before* decimals (also called “raw” amount or lamports for SOL)?15†L100-L107?.  

**Optional Query Parameters:**  
- **`slippageBps`** (integer): Slippage tolerance in basis points (e.g. 100 = 1% slippage allowed)?15†L101-L107?. If omitted, a default may be used.  
- **`platformFeeBps`** (integer): An integrator’s platform fee in basis points, if you wish to charge your users a fee?17†L182-L190?. If set, you must also provide a `feeAccount` when building the swap transaction to receive the fee?17†L182-L190?.  
- **`feeAccount`** (string): Token account address to receive the platform fee (if `platformFeeBps` is set)?17†L182-L190?.  
- **`restrictIntermediateTokens`** (boolean): If `true`, restricts routing so that intermediate hops use only highly liquid tokens (reducing risk of failure)?17†L168-L176?. Defaults to `false`.  
- **`onlyDirectRoutes`** (boolean): If `true`, only consider direct one-hop routes (single AMM)?17†L189-L197?. (Note: may result in no quote or worse pricing in many cases?17†L201-L204?.)  
- **`maxAccounts`** (integer): Limit the maximum number of token accounts (AMM accounts) used in the swap’s inner instructions?17†L208-L216?. This can help keep transactions within size limits when routing through many pools?17†L222-L230? (use with care – too low can exclude the best routes?17†L229-L237?).  
- **`asLegacyTransaction`** (boolean): If `true`, request the quote suitable for a legacy (v0) transaction instead of a versioned transaction with address tables?17†L175-L183?. Use this if the wallet doesn’t support versioned transactions (for compatibility)?17†L175-L183?.

**Example Request:** (Quote for swapping 1 SOL to USDC with 0.5% slippage, restricting intermediate tokens)  

```js
const quoteResponse = await (
  await fetch('https://lite-api.jup.ag/swap/v1/quote?inputMint=So11111111111111111111111111111111111111112&outputMint=EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v&amount=100000000&slippageBps=50&restrictIntermediateTokens=true')
).json();
console.log(JSON.stringify(quoteResponse, null, 2));
```?15†L113-L121?

**Response Structure:** On success, returns a JSON object containing the quote details:  
- **`inputMint`**, **`outputMint`**: Echo back the input/output mint addresses?15†L125-L133?.  
- **`inAmount`** (string): The input amount used (raw units)?15†L125-L133?.  
- **`outAmount`** (string): *Best* output amount achievable for the given input (raw units of output token)?15†L125-L133?. This assumes zero slippage impact; it is effectively the expected output if the price holds.  
- **`otherAmountThreshold`** (string): The minimum output amount considering the slippageBps. For an ExactIn swap, this is typically `outAmount * (1 - slippage)` (i.e., what Jupiter guarantees at execution or the swap will abort)?15†L127-L135?.  
- **`swapMode`** (string): `"ExactIn"` or `"ExactOut"` indicating whether the quote is for swapping a fixed input amount (ExactIn) or requesting a fixed output amount (ExactOut). (Swap API primarily uses ExactIn.)  
- **`slippageBps`** (integer): The slippage tolerance applied (in bps)?15†L131-L135?.  
- **`priceImpactPct`** (string): The estimated price impact percentage of this trade (as a decimal string)?9†L113-L121?.  
- **`platformFee`**: If a platform fee is applied, details of that fee (may be `null` if none)?15†L131-L134?.  
- **`routePlan`**: An array describing the swap route. Each element represents a segment of the route (one AMM swap), with fields:  
  - `swapInfo` (object) – details of the AMM/pool: `ammKey` (identifier of the pool program or market)?15†L135-L143?, `label` (name of the AMM, e.g. "Meteora DLMM")?15†L137-L145?, `inputMint`, `outputMint` for that segment, and `inAmount`, `outAmount` for that segment, plus any `feeAmount` and `feeMint` taken by that AMM?15†L135-L143?.  
  - `percent` (number) – what percentage of the trade flows through this route segment?15†L135-L143? (for multi-route splits). If a single route, this is 100.  
  - *(Note:* For a simple 1-hop route, `routePlan` will have one element at 100%. For multi-hop or split routes, multiple elements will be present.)*

**Notes:**  
- The `outAmount` in the quote is the best price *at the time of quote*. The slippageBps does **not** reduce this number; slippage is a tolerance for execution?17†L154-L161?. The actual executed amount will be at least `otherAmountThreshold` (accounting for slippage) or the swap will fail.  
- `restrictIntermediateTokens`, `onlyDirectRoutes`, and `maxAccounts` can influence route selection (potentially yielding a less optimal price)?17†L169-L177??17†L201-L204? – use these options only if needed (e.g. to avoid obscure tokens or transaction size issues).  
- If `maxAccounts` is used, note that it’s an *approximate* limit on the accounts used by the swap instructions only, not including setup/cleanup instructions?17†L222-L230?. ### POST `/swap/v1/swap` (Build Swap Transaction)

**Description:** Creates a **swap transaction** (unsigned) based on a quote and user wallet. This is a **transaction builder** – you supply the quote and any additional parameters, and Jupiter returns a compiled transaction (with all necessary instructions) ready to be signed and sent. This is the second step after obtaining a quote. The endpoint will return a base64 transaction that executes the swap according to the quote.

**Authorization:** Lite or Pro (with key).  
**Endpoint:** `POST https://lite-api.jup.ag/swap/v1/swap`?18†L53-L57?.

**Request Body (JSON):**  
- **`quoteResponse`** (object, required): The quote result object obtained from `/quote`. You can pass the entire quote JSON, or at minimum the fields it expects (typically the quote JSON is used directly)?18†L119-L127?.  
- **`userPublicKey`** (string, required): The user’s wallet public key (this will be the fee-payer and source/destination of funds in the transaction)?18†L125-L130?.  
- **Additional Optional Fields to optimize transaction landing**?18†L129-L137?:  
  - **`dynamicComputeUnitLimit`** (boolean): If `true`, the API will estimate and set an appropriate compute unit limit for the transaction (using Solana’s Compute Budget instruction)?18†L129-L137?. This can improve success rate by ensuring enough compute units.

### POST `/swap/v1/swap-instructions` (Advanced)

**Description:** An alternative to `/swap` that returns the **individual instructions** rather than a full transaction. **Authorization:** Same as `/swap` (Lite or Pro).

**Response Structure:** On success, returns a JSON object with fields:  
- **`setupInstructions`**, **`swapInstruction`**, **`cleanupInstruction`**: These are base64-encoded instructions (or structured data representing them) for the different phases of the swap?20†L223-L231? – e.g., setting up token accounts (ATA creation if needed), the main swap instruction, and post-swap cleanup (like unwrapping SOL if needed).

**Description:** Jupiter’s Swap API also provides an execute endpoint to send a **signed swap transaction** to the network (similar to Ultra’s execute). **Response:** Returns success or failure status with transaction signature and any error info (same structure as Ultra execute)?11†L163-L171??11†L182-L189?.

## Lend API (Jupiter Lend)

The **Jupiter Lend API** provides programmatic access to Jupiter’s lending/borrowing platform, including depositing assets to earn yield (“Earn”) and borrowing against collateral (“Borrow”). The Lend API interacts with on-chain programs (Jupiter’s lending programs) by building transactions for deposit, withdraw, etc. **Note:** As of mid-2025, the **Earn** API is in beta, and the **Borrow** API is work-in-progress (not yet fully released)?25†L39-L47??31†L39-L42?.

**Program IDs:** Jupiter Lend comprises multiple on-chain programs. For reference, the program IDs are: `LIQUIDITY_PROGRAM` (`jupeiUmn818Jg1ekPURTpr4mFo29p46vygyykFJ3wZC`), `LENDING_PROGRAM` (`jup3YeL8QhtSx1e253b2FDvsMNC87fDrgQZivbrndc9`), and others for risk management, oracle, and vaults?25†L41-L49?. ### Earn API (Deposit & Withdraw)

The Earn API allows users to **deposit assets** into Jupiter’s lending pools to earn yield, and **withdraw assets** from those pools. Each deposit yields a vault token (representing the user’s share in the pool). The API supports both operations specified by exact amount of underlying asset or by vault shares.

**Base URL:** `https://lite-api.jup.ag/lend/v1/earn/*`

#### POST `/lend/v1/earn/deposit`

**Description:** Creates a transaction for depositing a specific **amount of an asset** into the Earn protocol (lending pool). **Request Body:** (JSON)  
- **`asset`** (string, required): The mint address of the asset to deposit (e.g. USDC mint for depositing USDC)?29†L169-L177?.  
- **`amount`** (string, required): The amount of the asset to deposit, in raw base units (e.g. `100000` for 0.1 USDC if USDC has 6 decimals)?29†L169-L177?.  
- **`signer`** (string, required): The user’s wallet public key (this will be the source of funds and will need to sign the transaction)?29†L169-L177?.

## Trigger API (Limit Orders)

The **Trigger API** enables the creation of **limit orders** on Solana using Jupiter’s infrastructure?32†L45-L53?. A trigger order is an order to swap when certain conditions (price reaching a target) are met, effectively allowing limit buys/sells. These orders are created off-chain via Jupiter’s API and executed on-chain when triggered by Jupiter’s price monitor. The API provides endpoints to create, execute, cancel, and query these orders.

**Base URL:** `https://lite-api.jup.ag/trigger/v1/*` (Pro at `api.jup.ag/trigger/v1/*` with key).

### POST `/trigger/v1/createOrder`

**Description:** Creates a **new trigger (limit) order**. This returns an unsigned transaction for the order creation, which when signed and sent, places the limit order on Jupiter’s system. The order will wait until execution conditions are met (target price) and then execute automatically (by Jupiter’s backend). **Important:** In the latest API version, trigger orders are a two-step process: you *create* the order to place it, then *execute* it (either via API or manually) when conditions are met?33†L55-L63?. This two-step ensures the user signs the initial creation and later the execution.

**Authorization:** Lite or Pro (with key). Endpoint: `POST /trigger/v1/createOrder`?33†L49-L57?.

**Request Body:**  
- **`inputMint`** (string, required): Mint address of the input token (the token the user wants to swap *from* in the limit order)?33†L103-L111?.  
- **`outputMint`** (string, required): Mint address of the output token (the token the user will receive if the order executes)?33†L103-L111?.  
- **`maker`** (string, required): The user’s wallet address that will **hold the order** (maker account). In practice, this will be the same as the user’s address placing the order?33†L103-L111?.  
- **`payer`** (string, required): The address responsible for paying the transaction fee for order creation. Often the same as `maker` (user’s wallet)?33†L103-L111?.  
- **`params`** (object, required): Contains the core order parameters:  
  - **`makingAmount`** (string, required): The amount of `inputMint` to swap (raw units)?33†L107-L115?. This is the amount the user is offering to trade.  
  - **`takingAmount`** (string, required): The amount of `outputMint` the user wants in return (raw units)?33†L107-L115?. This effectively sets the limit price: the ratio `takingAmount`/`makingAmount` is the worst price the user is willing to accept. The order will only execute if the market can provide at least this much output for the given input.  
  - **`slippageBps`** (integer, optional): Slippage to allow on execution in bps. **By default (and on Jupiter UI for limit orders), this is 0** (Exact mode)?32†L95-L104??33†L83-L91? – meaning the order executes at the exact trigger price or better, with no additional slippage. If you set this to a value, you are effectively saying you’d allow some slippage beyond the limit price for potentially higher success; this corresponds to “Ultra” mode on the UI?32†L95-L104?. If omitted, assume 0 (Exact).  
  - **`expiredAt`** (integer, optional): An expiration time for the order in Unix timestamp (seconds)?33†L109-L117?. If set, the order will expire (become cancellable or not executable) after this time if not executed. If null or not provided, the order will not expire.  
  - **`feeBps`** (integer, optional): Referral fee in bps (if you as an integrator want to earn fees on the swap). If you include this, you **must** have a `feeAccount` set up (see below)?33†L89-L97?. Otherwise, omit or set to 0.  
- **`computeUnitPrice`** (string, optional): This controls the priority fee for execution. Setting `"auto"` will let Jupiter decide an optimal compute unit price (usually recommended)?33†L115-L120?. Alternatively, you could specify a value in micro-lamports/compute unit, but using `"auto"` is simplest.  
- **`feeAccount`** (string, optional): The referral token account for the **output mint** where your fee will be sent?33†L117-L120?. Required if `feeBps` is set. This account must exist and be registered in Jupiter’s referral program for the order to include the fee?33†L89-L97?.  
- **`wrapAndUnwrapSol`** (boolean, optional): Default `true`. If trading SOL, setting this to true will automatically wrap SOL to SPL WSOL and unwrap back to SOL for the user, simplifying SOL trades?33†L117-L120?. Usually keep this true.

**Note:** The Trigger API V1 is a new iteration (the older path was `/limit/v2` which is being deprecated)?33†L47-L55?. In V1:
  - There is now an explicit `/execute` endpoint (next section).  
  - `createOrder` now returns a `requestId` (to use with execute)?33†L55-L63?.  
  - The `cancelOrder` endpoint only cancels one order at a time, and a new `cancelOrders` exists to cancel multiple.  
  - The response field `tx` was renamed to `transaction` (or `transactions`)?33†L57-L64?.

**Example Request:**  

```js

  await fetch('https://api.jup.ag/trigger/v1/createOrder', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },

      inputMint: inputMint.toString(),
      outputMint: outputMint.toString(),

## Recurring API (Recurring Swap Orders)

The **Recurring API** allows creation of **recurring swap orders** – automated dollar-cost averaging (DCA) or value averaging strategies, where a user can schedule periodic swaps either on a time basis or triggered by price changes?36†L47-L55??36†L59-L67?. In simpler terms, a user can set up an order to, for example, buy $100 of SOL every day (time-based) or invest more when price drops (price-based), etc., and Jupiter’s infrastructure will execute these automatically over time.

**Base URL:** `https://lite-api.jup.ag/recurring/v1/*` (Pro at `api.jup.ag/recurring/v1/*`).

Two types of recurring orders are supported: **Time-based** (execute at regular intervals) and **Price-based** (execute when price moves by certain increments). The API uses a unified `createOrder` endpoint for both; you specify parameters for either time or price strategy.

### POST `/recurring/v1/createOrder`

**Description:** Creates a **recurring order** (either time-based or price-based) and returns an unsigned transaction to set up that order?37†L60-L68?. The user must sign and send this transaction to initiate the recurring strategy. The order will then live on Jupiter’s system and execute over time.

**Authorization:** Lite/Pro.

**Request Body:**  
- **`user`** (string, required): The user’s wallet address (funds will be taken from this wallet for each swap)?37†L98-L105?. This user will also receive the outputs of each swap.  
- **`inputMint`** (string, required): Mint address of the token to spend in each swap cycle?37†L98-L105?.  
- **`outputMint`** (string, required): Mint address of the token to acquire in each swap cycle?37†L98-L105?.  
- **`params`** (object, required): Contains either a `time` object or a `price` object, determining the strategy:  

  **Time-based strategy (`params.time`):**?37†L71-L79?  
  - **`inAmount`** (string, required): The total amount of input token the user plans to invest over the entire plan (raw units)?37†L101-L109?. This amount will be split across the recurring swaps.  
  - **`numberOfOrders`** (integer, required): The total number of swap executions to perform?37†L101-L109?.  
  - **`interval`** (integer, required): Time between each order in seconds?37†L105-L112?. e.g., `86400` for daily, `604800` for weekly, etc.  
  - **`minPrice`** (string or null, optional): A minimum price threshold (in terms of output per input) – if the price goes below this, perhaps skip or adjust? (The documentation mentions it, but likely if provided, it might not execute below this price – effectively a safeguard)?37†L107-L112?. It can be `null` for no minimum price condition.  
  - **`maxPrice`** (string or null, optional): A maximum price threshold – if price goes above this, maybe skip execution. `null` for no max?37†L107-L112?.  
  - **`startAt`** (integer or null, optional): Unix timestamp when to start the first swap?37†L109-L113?. If `null`, start immediately upon order creation. If a timestamp is given, the first swap will wait until that time.  

  *How it works:* The API will calculate **amount per order** = `inAmount / numberOfOrders`?37†L79-L87?. This fixed chunk will be swapped each interval. The total time to complete = `numberOfOrders * interval`?37†L83-L90? (so essentially after that many intervals the plan ends). The user needs to have `inAmount` ready (either all at once or deposit per cycle – Jupiter likely takes the `inAmount` into an escrow and then uses it per cycle).

  **Price-based strategy (`params.price`):**?37†L119-L127?  
  - **`depositAmount`** (string, required): The initial amount of input token to deposit now into the recurring order (raw units)?37†L149-L157?. This sets up the pool of funds to use for recurring buys.  
  - **`incrementUsdcValue`** (string, required): The increment in USD value for each cycle?37†L151-L158?. Essentially, how much more USD worth of the input token to spend each time the price condition triggers. For example, if you set 10 USDC (in raw units, e.g. `10000000` for 10.0) as increment, every time the price condition is met, the order will spend an amount of input token worth an additional 10 USDC. This is used for value averaging strategies (buy more when price drops to keep value increasing by fixed increments).  
  - **`interval`** (integer, required): Minimum time between each price-based execution, in seconds?37†L153-L158?. Even if price moves rapidly, it won’t execute more often than this interval. E.g., 86400 for at most once per day.  
  - **`startAt`** (integer or null, optional): Start time for the strategy (same idea as time-based, null to start immediately)?37†L155-L158?.

  *How it works:* Price-based orders are **open-ended** (no fixed number of orders)?37†L127-L134?. They will continue until you cancel or run out of funds. Each time the output token’s price (relative to input) changes such that the value of input held has dropped by `incrementUsdcValue` (or however the strategy is defined), the order will execute to invest that incremental amount. Jupiter automatically withdraws output tokens to the user’s wallet on each execution?37†L131-L138?. The user can deposit more or withdraw input funds at any time using the deposit/withdraw endpoints (below) to extend or pull out of the strategy.

- A withdraw will return the specified `amount` to the user and reduce the funds in the order. If the `amount` equals all remaining funds and `inputOrOutput` is "In", effectively the user has pulled out all unspent funds but the order might technically still exist (perhaps needing cancellation to fully remove, unless an empty order auto-expires).

**Note:** `priceWithdraw` with `inputOrOutput: "Out"` would withdraw any held output tokens in the order. However, as mentioned, in V3 Jupiter auto-withdraws output each execution, so usually there are no accumulated output tokens to withdraw (hence the focus is on "In").

### GET `/recurring/v1/getRecurringOrders`

**Description:** Fetches recurring orders for a user, similar to Trigger’s getTriggerOrders. It lists the user’s active recurring orders or past ones.

**Query Parameters:**  
- `user` (string, required): User’s wallet address?44†L57-L65?.  
- `orderStatus` (string, required): `"active"` for ongoing orders, `"history"` for completed/ended orders?44†L63-L71?.  
- `recurringType` (string, optional): `"time"`, `"price"`, or `"all"` – filter by type or get all?44†L63-L68?.  
- `includeFailedTx` (boolean, optional): Include any failed transactions in history if `true`?44†L65-L68?.  
- `page` (integer, optional): Page number for pagination (if many orders).

**Example:**  

```js
const activeRecurring = await (
  await fetch('https://lite-api.jup.ag/recurring/v1/getRecurringOrders?user=YourWallet&orderStatus=active&recurringType=all')
).json();
```?44†L79-L87?

**Response:** Likely returns an array of orders or an object with order data. Each order entry would include whether it’s time or price, details of the schedule or increments, how much has been executed so far, how much remains, etc., as well as identifiers for the order. For history entries, it might show final status (completed, canceled) and any leftover funds returned.

For instance, an active time-based order might show: `{ type: "time", inputMint: ..., outputMint: ..., totalAmount: ..., amountPerOrder: ..., executed: X of N orders done, nextExecutionAt: ..., ... }`. A price-based might show current deposit remaining, increment, etc.

*(Since no explicit JSON is given, treat it as a query interface to show users their plans.)*

**Best Practices & Notes:**  
- **Fees:** Currently, Recurring API does **not support integrator fees** on swaps?36†L87-L93?. So you cannot charge a referral fee on recurring orders (Jupiter’s own fee of 0.1% still applies on each swap though).  
- **Monitoring:** Recurring swaps run automatically. Jupiter charges 0.1% fee on each swap executed via Recurring API?36†L87-L95?.  
- **Cancellation vs Withdraw:** If a user wants to stop a price-based plan but maybe leave funds for later, they could withdraw funds (`priceWithdraw`) without canceling, effectively pausing the activity (because no funds to execute). But to truly stop it, use cancelOrder. For time-based, cancel if they want to stop early.  
- **Partial completion:** Time-based orders naturally end after the set number of cycles. Price-based orders go indefinitely until canceled.  
- **Notifications:** As an integrator, you may want to inform users to ensure they have sufficient funds in the order. Jupiter’s docs note that price-based orders won’t close automatically when funds run out; they’ll just pause. Users can deposit more to continue or cancel. Also, if price-based orders have output auto-withdrawn, users will see output tokens appear in their wallet after each execution.  

## Token API

The **Token API** provides endpoints to query token metadata, perform token verification, and retrieve token lists. It’s essentially Jupiter’s token discovery and info service. The latest version is **V2** (Beta) with enriched data, while **V1** is deprecated as of August 2025?48†L47-L55?.

**Use Cases:** Validate if a mint is a legitimate token, fetch token name/symbol/icon, get lists of verified tokens, trending tokens, etc., which can be used to populate token pickers or show token info in your app.

### Token API V2 (current)

**Base URL:** `https://lite-api.jup.ag/tokens/v2`

V2 offers multiple endpoints: **search by query**, **filter by tag**, **get category**, **get recent** tokens. All endpoints return token data objects with fields described below.

#### GET `/tokens/v2/search?query=<term>`

**Description:** Search tokens by symbol, name, or mint. Returns up to 20 results by default for text queries?49†L79-L87??49†L81-L89?, or up to 100 if querying multiple specific mint addresses. This is similar to Ultra’s search but can be used standalone.

**Query Param:** `query` – string to search. Can be a full or partial token symbol/name, or one or multiple mint addresses (comma-separated)?49†L79-L87?.

**Example:**  

```js
const results = await (await fetch('https://lite-api.jup.ag/tokens/v2/search?query=SOL')).json();

    This might return tokens like SOL, mSOL, etc.

    (Response format and fields are discussed under “Response Fields” below.)

    
    GET /tokens/v2/tag?query=<tag>

    Description: Get all tokens that have a specific tag[28]. Tags classify tokens (e.g., verified, lst, etc.).

    Query Param: query – the tag name. Supported tags in V2 include: - verified – community/Jupiter verified tokens[29],
- lst – liquid staked tokens list[29],
- token-2022 – tokens using the Token-2022 program,
- potentially others like community or strict if defined internally.

    This returns an array of token objects matching that tag (could be large; e.g., verified returns all verified tokens).

    Example:

    js const verifiedTokens = await (await fetch('https://lite-api.jup.ag/tokens/v2/tag?query=verified')).json();[30]

    
    
    GET /tokens/v2/<category>/<interval>?limit=<N>

    Description: Get top tokens by certain categories over a time interval[31]. Categories include: - toporganicscore – tokens with highest Organic Score (an indicator of genuine activity)[32], - toptraded – most traded tokens, - toptrending – tokens with big recent increases in activity.

    Path Parameters:
- <category>: one of toporganicscore, toptraded, toptrending[33].
- <interval>: one of time intervals 5m, 1h, 6h, 24h[33]. This interval defines the window over which “top” is measured (except for organic score which might be more static but possibly updated periodically).

    Query Param: limit – how many results to return (default 50 if not given)[34]. You can adjust to get more or fewer.

    Example:

    js const topOrganic = await (await fetch('https://lite-api.jup.ag/tokens/v2/toporganicscore/24h?limit=100')).json();[35]

    This fetches the top 100 tokens by organic score in the last 24h (with common tokens like SOL/USDC filtered out as note suggests[36]).

    (Note: The API might filter out SOL, USDC etc. from trending categories to surface more interesting tokens[33].)

    
    
    GET /tokens/v2/recent

    Description: Get a list of recently listed tokens[37]. “Recent” is defined by when the token’s first liquidity pool was created (not just when the mint was made)[38]. This is useful to find new tokens that have just launched/traded.

    Query Param: None for basic use (you can possibly use limit similarly; by default it returns 30)[38].

    Example:

    js const newTokens = await (await fetch('https://lite-api.jup.ag/tokens/v2/recent')).json();[39]

    By default, it returns up to 30 new tokens.

    
    
    Token Data Fields (V2 Response)

    All these V2 endpoints return an array of token info objects[40][41]. Each object can include:
- id: token mint address[42].
- name: token name[42].
- symbol: token symbol/ticker[43].
- icon: URL of token icon image[43].
- decimals: number of decimals for token[43].
- circSupply and totalSupply: circulating and total supply (as floating numbers)[44].
- tokenProgram: which SPL program it’s under (usually the standard Token program, or Token-2022 program)[44].
- firstPool: an object with id (address of first liquidity pool) and createdAt (ISO timestamp) for that pool[45]. This indicates when/where the token first became liquid.
- holderCount: number of holders addresses (approximate)[46].
- audit: an object with certain token audit info: mintAuthorityDisabled (bool), freezeAuthorityDisabled (bool), topHoldersPercentage (the percent of supply held by top holders)[46]. These help assess centralization/trust (e.g., if mint and freeze authorities are renounced).
- organicScore: numeric score (0-100) for organic activity[47].
- organicScoreLabel: e.g. "high", "medium", "low" etc., giving a qualitative label to the organic score[47].
- isVerified: boolean indicating if token is verified by Jupiter’s verification process[48].
- cexes: array of centralized exchanges where this token is listed (if known)[49]. E.g., ["Binance","Kraken",...] or empty if none.
- tags: array of tags assigned (like "community", "verified", "lst", etc.)[50].
- fdv: Fully Diluted Value (market cap if all supply is in circulation)[51].
- mcap: current Market Cap (circulating supply * price)[51].
- usdPrice: current price in USD[51].
- priceBlockId: the block ID (slot or block number) at which the price was last updated[52].
- liquidity: current on-chain liquidity (in USD) across Jupiter’s pools for this token[53].
- stats5m, stats1h, stats6h, stats24h: objects containing stats over those intervals[24][25]. Each includes fields like: priceChange (fractional change in price), liquidityChange (change in liquidity), volumeChange (change in volume) – these likely as percentages or factors; and buyVolume, sellVolume (total volume bought/sold), buyOrganicVolume, sellOrganicVolume (volume excluding wash trades perhaps), numBuys, numSells, numTraders, numOrganicBuyers, numNetBuyers (how many more buyers than sellers)[24][25]. These give a detailed picture of trading activity.
- ctLikes and smartCtLikes: perhaps crowd sentiment metrics (maybe number of social media likes or something from “CoinTalk”? – Not entirely clear, but they have values e.g. 4232 and 522 in example for SOL)[54]. Possibly related to social sentiment.
- updatedAt: timestamp of last data update for this token entry[54].

    This is a superset of fields; not all tokens or endpoints might return everything (e.g., tags might be empty array if none). But this is what to expect.

    Example Response (excerpt):

    json [ { "id": "So11111111111111111111111111111111111111112", "name": "Wrapped SOL", "symbol": "SOL", "icon": "https://.../So111.../logo.png", "decimals": 9, "circSupply": 531207433.3986673, "totalSupply": 603724547.3627878, "tokenProgram": "TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA", "firstPool": { "id": "58oQChx4yWmvKdwLLZzBi4ChoCc2fqCUWBkwMihLYQo2", "createdAt": "2021-03-29T10:05:48Z" }, "holderCount": 2342610, "audit": { "mintAuthorityDisabled": true, "freezeAuthorityDisabled": true, "topHoldersPercentage": 1.2422471238911812 }, "organicScore": 98.92390784896082, "organicScoreLabel": "high", "isVerified": true, "cexes": ["Binance","Bybit","OKX","Upbit","Bitget","Kraken","KuCoin","MEXC","Gate.io"], "tags": ["community","strict","verified"], "fdv": 87824499429.22047, "mcap": 77275352037.79674, "usdPrice": 145.47114211747515, "priceBlockId": 349038717, "liquidity": 89970631.83880953, "stats5m": { "priceChange": 0.02117, "liquidityChange": -0.01230, "volumeChange": 4.8551, "buyVolume": 14644327.1883, "sellVolume": 14743625.0239, ... }, "stats24h": { "priceChange": 1.5076, "liquidityChange": 2.4173, "volumeChange": -2.1516, "buyVolume": 4273248565.2568, "sellVolume": 4306065610.69747, ... }, "ctLikes": 4232, "smartCtLikes": 522, "updatedAt": "2025-06-25T05:02:21.034234634Z" } ][41][49]

    (This example is for SOL from an earlier section, but demonstrates the richness of data V2 provides.)

    
    
    Token API V2 Reference & Changes

    V2 simplifies output to a single price (usdPrice) instead of multiple prices like V1 did[55]. It intentionally provides one authoritative price per token to avoid confusion[56].

    Some data from V1 (like separate buy/sell price, confidence, etc.) are not present in V2 directly. For more detailed pricing info (like depth or confidence), one should use the Price API or the Swap quote endpoint[57] (or see the note referencing a research post about how V2 was derived[58]).

    The Organic Score is a new metric in V2 to help gauge how "real" a token’s activity is[59].

    
    
    
    Token API V1 (deprecated)

    Base URL: https://lite-api.jup.ag/tokens/v1 – This version is deprecated by Aug 1, 2025[27]. New integrations should use V2.

    For historical context or if needed:

    V1 had endpoints:
- GET /tokens/v1/token/<mint>: Fetch token info by mint[60]. Returns basic info (address, name, symbol, logoURI, tags, creation time, authorities, etc.)[61][62]. (The example shows Jupiter token details, including tags and extension like coingeckoId)[61][63].
- GET /tokens/v1/market/<marketAddress>/mints: Given a liquidity pool/market address, returns the two token mints involved in that market[64][65].
- GET /tokens/v1/mints/tradable: Returns a list of all token mints that are tradable on Jupiter (passed Jupiter’s liquidity checks)[66][67]. This essentially lists all tokens you can get a quote for.
- GET /tokens/v1/tagged/<tag>: Returns a list of token mints (with some info) that have a given tag[68][69]. In V1, the format might include the token info objects or just list of mints. The example shows usage of tag lst (liquid staked tokens)[70]. Possibly multiple tags could be combined by comma, as hint suggests (in V1 you could do that)[71].

    V1’s output was simpler. For example, /token/<mint> gave one object with fields such as: json { "address": "...", "name": "...", "symbol": "...", "decimals": 6, "logoURI": "...", "tags": [ "...", "..." ], "daily_volume": ..., "created_at": "...", "freeze_authority": "...", "mint_authority": "...", "permanent_delegate": "...", "minted_at": "..." , "extensions": { "coingeckoId": "..." } }[61][63]

    Lists like /mints/tradable or /tagged/... returned arrays of either mint addresses or objects. E.g., tradable returned an array of mint addresses[72][73]. Tagged likely returned objects (the guide suggests it returns "token mints (with information)" for tagged tokens[68]).

    Important: V1 is deprecated. It had fewer data points and required more client-side work to interpret (e.g., separate endpoint to check tags). V2 consolidates and extends these. If you’re using V1, plan migration to V2.

    

    
    
    
    Price API

    The Price API provides token price data for tokens on Solana. It aims to be a single source of truth for token prices across Jupiter and integrator platforms[74]. It considers on-chain swap prices and oracle prices for reference assets to produce a reliable USD price for each token.

    
    Price API V3 (current, Beta)

    Base URL: https://lite-api.jup.ag/price/v3 (Pro: api.jup.ag/price/v3)

    V3 simplifies and improves the Price API by focusing on last traded price and applying various heuristics to ensure accuracy[75]. It returns a single price (USD) per token, rather than the richer but sometimes confusing data of V2.

    Endpoint: GET /price/v3?ids=<mint1>,<mint2>,...

    Query Parameters:
- ids (string, required): Comma-separated list of token mint addresses for which to retrieve prices[76]. You can request multiple at once (up to 50 mints per query)[77].

    Example Request:

    js const prices = await (await fetch('https://lite-api.jup.ag/price/v3?ids=So11111111111111111111111111111111111111112,JUPyiwrYJFskUPiHa7hkeR8VUtAeFoSYbKedZNsDvCN')).json();[78]

    Response Structure: Returns a JSON object where keys are the mint addresses requested, and values are objects containing:
- usdPrice (number): The price of the token in USD[79].
- blockId (integer): The slot or block number as of which this price is computed[80]. Use this to know how recent the price is.
- decimals (integer): The token’s decimals (useful to interpret other data or for formatting)[80].
- priceChange24h (number): The 24-hour price change percentage (as a decimal fraction)[81]. This indicates how much the price moved in the last day.

    It does not provide separate bid/ask or depth – just the price. Jupiter derived this by starting from known reliable tokens (like SOL via Pyth or Switchboard or etc.) and propagating prices through swap pools, with filters for manipulation[82].

    Example Response:

    json { "JUPyiwrYJFskUPiHa7hkeR8VUtAeFoSYbKedZNsDvCN": { "usdPrice": 0.4056018512541055, "blockId": 348004026, "decimals": 6, "priceChange24h": 0.5292887924920519 }, "So11111111111111111111111111111111111111112": { "usdPrice": 147.4789340738336, "blockId": 348004023, "decimals": 9, "priceChange24h": 1.2907622140620008 } }[79][81]

    (Above: JUP token at ~$0.4056, and SOL at ~$147.4789, along with their 24h changes and data block.)

    Characteristics of V3:
- It gives one price per token (USD). This price is the last swapped price on Solana, anchored to reliable tokens like SOL which use external oracle input[75]. Essentially, Jupiter tracks swaps and uses a core set of trusted tokens to compute others.
- It applies heuristics to avoid spoofed prices (ignoring wash trades, tiny pools, etc.)[83][84]. - The priceChange24h is useful to show trend. If a token hasn’t traded in 7 days, it likely won’t have a price (the API might not list it)[85].
- If a token is flagged as unreliable (low liquidity, potential manipulation, etc.), Price API might not provide a price at all to prevent showing a wrong number[85]. In such cases, the response just won’t have that token, or you’ll know it can’t be priced reliably.

    Limitations:
- You can query up to 50 tokens per call[77].
- If you query a token with no recent trading activity (last 7 days) or which is flagged (by heuristics) as too risky, you may get no price for it[85].
- The V3 focuses on USD price. If you need more info like volumes or multiple quote currencies, consider using Swap API’s quote for specific pairs or refer to historical data.

    Comparison to V2: V2 (deprecated) provided more detail: it had buy and sell prices, confidences, depth info, etc., but it was complex and sometimes led to different interpretations of price[86]. V3 streamlines this by doing the heavy logic internally and outputting one price. If you need something like mid-price vs last price, Jupiter suggests using the Swap API’s /quote which effectively gives you an actionable price (with slippage etc.), or refer to Jupiter’s research post on how V2 was derived if needed[58].

    
    
    Price API V2 (deprecated)

    Base URL: https://lite-api.jup.ag/price/v2

    V2 provided extensive price info, including “extraInfo” with depth and confidence. It’s deprecated by August 2025 in favor of V3[87].

    Key endpoints in V2 were:
- GET /price/v2?ids=<mint1>,<mint2>,... (with optional vsToken=<mint> and showExtraInfo=true parameters)[88][89].
- By default, prices were quoted vs USDC (if no vsToken given)[89]. It returned a JSON with a "data" field containing each requested token’s price info.
- If vsToken specified, it would price relative to that token instead of USD (e.g., get SOL price in terms of SRM, etc.)[90][91].
- If showExtraInfo=true, the response included rich info: both buy/sell prices, last swap price, current quote price, confidence, depth, etc.[92][93].

    An example basic response (with no extra info, vs USDC) looked like:

    json { "data": { "<Mint1>": { "id": "<Mint1>", "type": "derivedPrice", "price": "210.195311500" }, "<Mint2>": { "id": "<Mint2>", "type": "derivedPrice", "price": "1.084247" } }, "timeTaken": 0.00488491 }[94][95]

    This shows the derived price (likely mid-price) of two tokens vs USDC.

    With showExtraInfo=true, the data object for each token became much larger, e.g.:

    json { "id": "<Mint>", "type": "derivedPrice", "price": "...", "extraInfo": { "lastSwappedPrice": { "lastJupiterSellAt": <timestamp>, "lastJupiterSellPrice": "...", "lastJupiterBuyAt": ..., "lastJupiterBuyPrice": "..." }, "vwap": { ... }, "currentPrice": { "buy": "...", "sell": "...", "mid": "...", ... }, "confidence": "...", "depth": { "buy": "...", "sell": "..." }, "etc...": "..." } }[96][97]

    It included buy/sell prices, volumes, etc. But as mentioned, this caused different platforms to use different fields as "the price", leading to inconsistency[86]. Thus V3’s simplification.

    Migration: For current integrations, use Price API V3 for a straightforward price. If more detail is needed (like order book depth or multiple quote currencies), one might combine Jupiter’s data with an external source or use the Swap API to simulate trades of various sizes for effective price and slippage.
