Strategy Overview
SolanaCrossDexTriangle is a cross-exchange triangular arbitrage strategy designed for the TradeMonitor platform. Its purpose is to exploit price discrepancies between three different tokens across multiple Solana DEXs, executing a three-leg loop (Token A → Token B → Token C → back to Token A) for profit. In traditional triangular arbitrage, all three trades occur on one exchange
moonpay.com
. In contrast, SolanaCrossDexTriangle extends this concept across multiple decentralized exchanges on Solana (e.g. Raydium, Orca, Jupiter). This broader scope allows the strategy to capture inter-exchange inefficiencies that a single-DEX strategy might miss. The expected use case is an algorithmic trading bot continuously scanning Solana DEX markets for triangular cycles that yield a net gain in the starting asset after accounting for fees and slippage. This strategy is particularly useful for advanced DeFi arbitrageurs aiming to maximize returns from Solana’s vibrant multi-DEX ecosystem. Use Case Example: A trader running SolanaCrossDexTriangle might find an opportunity where SOL, USDC, and BONK prices are misaligned across Raydium, Orca, and Jupiter. For instance, starting with SOL on Raydium, swapping SOL → USDC, then using Orca to swap USDC → BONK, and finally swapping BONK → SOL on Jupiter could result in more SOL than initially started (if such price differences exist). The strategy automates detection of such cycles and executes them faster than a human could, aiming to lock in risk-free (or minimal risk) profit.

Configuration & Adjustable Parameters
SolanaCrossDexTriangle offers a range of runtime-configurable parameters to fine-tune its behavior. These parameters can be adjusted via configuration files or UI at runtime, without code changes, allowing the strategy to be tailored to different risk appetites and market conditions. Key configuration settings include:
MinimumProfitThreshold: Profit Floor for Execution. This represents the minimum expected profit required for an arbitrage cycle to be considered worthwhile. It is typically expressed as a proportion of the starting amount (e.g. 0.005 for 0.5% profit). The strategy will ignore any triangular opportunity unless the estimated net gain (after fees and slippage) exceeds this threshold. This guards against trivial or potentially negative-profit trades once transaction costs are considered.
MaxTradeAmount: Upper Trade Size Limit. This parameter caps the amount of the starting token that the strategy will use in any arbitrage loop. Even if a cycle appears profitable, the strategy will not trade more than this amount in one go. This helps limit exposure and slippage; large trades can move prices significantly, so capping trade size is a risk control. MaxTradeAmount can be set based on liquidity considerations (e.g., a low-liquidity token should use a smaller max trade to avoid too much price impact).
MinTradeAmount: Lower Trade Size Limit. This is the smallest trade amount the strategy will attempt. It prevents execution of very tiny arbitrages that would yield negligible absolute profit or be lost to fees. For example, if set to 10 USDC, the strategy won’t bother executing cycles that start with less than 10 USDC worth of value. This ensures efficient use of resources and avoids wasting transaction fees on dust trades.
MaxExecutionTimeSeconds: Timeout for Cycle Execution. This defines the maximum allowed time (in seconds) to complete the full three-leg arbitrage transaction. If the sequence of trades cannot be completed within this time (due to network latency, blockchain congestion, etc.), the opportunity is considered too risky (prices might change) and should be skipped. In practice, if using atomic transactions, this might be just a few seconds (since it’s one transaction). If sequential, the strategy/bot will enforce a timeout: for instance, if after X seconds the cycle isn’t done, it might cancel remaining legs. This parameter helps mitigate execution risk by avoiding stale trades.
SlippageTolerance: Allowed Price Slippage. This setting (often a percentage) indicates how much worse the execution price is allowed to be compared to the initial quote before the trade is aborted. Each leg of the triangle can move in price by the time the trade executes; slippage tolerance ensures we account for that. For example, a 0.5% slippage tolerance means if any leg’s price slips more than 0.5% from the expected rate, the trade (or that leg) should not execute (or should fail fast) to avoid turning a profit into a loss. This effectively sets the acceptable price deviation threshold

. The strategy uses this in simulation (see Trade Simulation below) to only consider opportunities that remain profitable even with slippage, and it will also configure the actual swap orders with limit prices or parameters to enforce this tolerance (especially if using an aggregator or on-chain instruction, e.g. setting the minimum acceptable output for each swap).
SupportedTokens: Whitelist of Tokens to Consider. A list of token symbols (or IDs) that the strategy will include in its search for cycles. By restricting to a known set of tokens (e.g. ["SOL", "USDC", "USDT", "BONK", "wETH"]), we limit the search space and avoid illiquid or unwanted assets. Typically, this list would include major base tokens and popular SPL tokens where arbitrage is plausible. The tokens here should also correspond to holdings the bot can trade or assets configured in the wallet. This list is adjustable at runtime – adding or removing tokens changes which triangles are scanned.
SupportedExchanges: Which DEXs to Use. This is a list of Solana exchanges on which the strategy will look for price differences. For example: ["RAYDIUM", "ORCA", "JUPITER"]. Only pools/routes on these exchanges are considered when building the arbitrage cycles. By tuning this list, one can include more DEXs (increasing opportunities but also computational load) or exclude certain DEXs (perhaps due to reliability issues or lack of API access). The default focus is on Solana DEXs; Jupiter (as an aggregator) might be included as a pseudo-exchange for one of the legs. All exchanges listed here must have corresponding IExchangeService implementations in TradeMonitor (Raydium, Orca, etc. are provided out-of-the-box).
MaxConcurrentTrades: Parallel Trade Limit. This setting controls how many arbitrage cycles the strategy (or the bot) can execute simultaneously. A value of 1 means the strategy will only undertake one triangle arbitrage at a time, waiting for it to complete before starting another. Higher values (2, 3, etc.) allow multiple independent opportunities to be executed concurrently if found. Concurrency can improve profit capture in very active markets, but it also increases complexity and risk (e.g. managing multiple ongoing swaps and higher combined exposure). Typically, for arbitrage, it’s safest to do one at a time (to ensure full attention and capital is available), so MaxConcurrentTrades might default to 1 in most configurations. If increased, the strategy/bot will enforce that no more than that number of trade commands are active in the CommandChannel at once.
All these parameters are defined in the strategy’s configuration class (implementing IStrategyConfiguration) and can be injected or reloaded from config at runtime. They are exposed to users so that one can adjust strategy behavior without recompiling – for instance, raising the MinimumProfitThreshold in volatile times to only catch larger arbitrages, or expanding SupportedTokens if new tokens become liquid.
Risk Profiles (Low, Medium, High)
To accommodate different risk appetites, the strategy can be run under preset risk profiles – typically Low, Medium, and High risk. Each profile tweaks the configuration parameters and internal safeguards to balance profit potential against safety. Below we outline how the profiles differ:
Low Risk: Prioritizes capital preservation and high-confidence trades. In this mode, the strategy sets a higher MinimumProfitThreshold (e.g. require >1% profit) to ensure a substantial cushion on any trade

. SlippageTolerance is very tight (small, e.g. 0.1-0.2%) so that even minor adverse price moves will prevent execution – this avoids accidentally losing money if prices shift. MaxTradeAmount is conservative (small trade sizes) to minimize impact and exposure, and MinTradeAmount might be higher than default to avoid many small risky trades. Additionally, Low Risk profile would enforce MaxConcurrentTrades = 1 (only one trade at a time, no simultaneous positions) and might even introduce delays between trades or extra confirmations. The focus is on quality over quantity: only obvious, low-risk arbitrage loops are executed. This profile suits volatile markets or when using large capital that needs to be safeguarded.
Medium Risk: A balanced approach between safety and opportunity. The MinimumProfitThreshold might be moderate (e.g. 0.5%), allowing more opportunities through than low risk, but still ensuring some buffer. SlippageTolerance is set to a moderate level (e.g. 0.5%), accepting a bit of price movement risk in exchange for not missing out on trades. MaxTradeAmount is set to a reasonable fraction of typical pool liquidity – larger than low risk but still mindful of not swinging the market too much (for example, if Low risk allowed 100 SOL max, Medium might allow 200 SOL). MaxConcurrentTrades could be 1 or possibly 2 if the operator is comfortable handling two at once, but usually it remains 1 to avoid overlap. Medium risk profile will execute opportunities that have a solid expected profit and manageable risk, and is suitable for normal market conditions.
High Risk: Focuses on maximizing profit and capturing even small arbitrage gaps, at the cost of higher potential drawdowns. MinimumProfitThreshold is low (e.g. 0.1% or even just enough to cover fees), meaning the strategy will attempt very fine margin trades. SlippageTolerance is wider (perhaps 1% or more), acknowledging that some trades might slip and still be worth it – the strategy is willing to risk that a portion of the profit might erode or even break-even in hopes that many fast trades yield net profit. MaxTradeAmount is larger, utilizing more of the available capital and liquidity (as long as it's within exchange limits) – this can increase profits per trade but also increases market impact. Additionally, MaxConcurrentTrades might be higher in this profile (e.g. 2 or 3), allowing multiple cycles to run in parallel if multiple distinct opportunities arise (for example, arbitraging different token triangles simultaneously). The High risk profile expects the operator to closely monitor performance and possibly use advanced features like no stop-loss (since the strategy expects to exit each trade immediately by design; a failed arb is effectively a loss if it happens). It may also disable certain circuit breakers unless configured, trusting the strategy’s rapid execution to get out before trouble. This profile is aggressive and is typically used only in stable market conditions or with smaller capital that one is willing to put at greater risk for higher returns.
Internally, these profiles can be realized by providing different sets of default parameter values and risk guard toggles. For example, TradeMonitor’s configuration might allow specifying "RiskProfile": "Low" under the strategy settings, which the strategy reads and accordingly scales the parameters (or one can simply manually set the parameters to achieve the desired profile). Moreover, TradeMonitor’s global risk management features (like circuit breakers or max position limits

The strategy itself remains the same algorithm, but these profile-based settings drastically alter its behavior to match the user’s risk tolerance.
Arbitrage Cycle Detection Logic
Detecting profitable cycles across multiple exchanges is a core challenge of the SolanaCrossDexTriangle strategy. The strategy employs a graph-based search to efficiently identify triangular arbitrage opportunities (though a brute-force triple nested loop is conceptually possible, graph algorithms scale better for many tokens/exchanges). The detection logic operates roughly as follows:
Graph Construction: The strategy builds or updates a directed graph where nodes represent tokens and edges represent available swap paths on a particular exchange. If exchange X offers a trading pair (TokenA, TokenB), we create two directed edges: A → B (via X) and B → A (via X). Each edge carries a weight that represents the current conversion rate (price) from the source token to the target token on that exchange, factoring in fees. For AMM DEXs like Raydium/Orca, this rate can be derived from the pool’s reserve ratios and fee structure. For Jupiter (aggregator), an edge from TokenA to TokenB would represent the best rate Jupiter can find to swap A to B at the moment (though Jupiter internally might route through multiple pools, we treat it abstractly as one “hop” for A→B with a given quote).
Cycle Search: With this multi-exchange liquidity graph, the strategy searches for cycles of length 3 that start and end at the same token (A → B → C → A). In graph terms, we look for simple cycles of length 3 (triangles) in the directed graph. There are various ways to do this:
A brute-force approach would iterate over all triples of distinct tokens (A, B, C) in SupportedTokens, and for each, check if edges exist A→B on some exchange, B→C on some (possibly different) exchange, and C→A on another. If all three edges exist, you have a candidate cycle. This is straightforward but can be inefficient if token list is large.
A more efficient graph approach is to traverse neighbors: for each token A (as start), look at all possible B such that an edge A→B exists; then for each such B, look at all C such that an edge B→C exists; if there’s an edge C→A, we found a cycle. This avoids some redundant checks and leverages adjacency lists.
One can also use algorithms like Bellman-Ford or repeated DFS to detect positive-weight cycles in a weighted graph. A common method is to transform exchange rates to log-space and find cycles where the sum of log weights is positive (indicating product of rates > 1)
medium.com
github.com
. However, since we only want 3-leg cycles (not longer paths), a targeted triple search is sufficient.
Candidate Filtering: Not every cycle in the graph is profitable; many will yield less than 1x after fees. For each detected cycle A→B→C→A, the strategy computes the cumulative conversion rate: essentially multiply the rate A→B (on exchange X), by B→C (on Y), by C→A (on Z). If the resulting product > 1, it means 1 unit of A would end up as more than 1 unit of A after the three swaps – an arbitrage exists. We further require it to exceed (1 + MinimumProfitThreshold) to account for desired profit. If product after subtracting fees is <= 1, it’s not profitable. This check is done immediately during the search to prune non-profitable cycles quickly (this corresponds to the “Initial Screening” in strategy execution flow).
Cycle Selection: If multiple profitable cycles are found, the strategy may rank them (for example, by highest expected ROI or by absolute profit given a certain trade amount). It could also consider other criteria like liquidity (a cycle might show a profit for a tiny trade but not scale to a useful size). Typically, the strategy would choose the single best opportunity to act on at any given moment (since executing one might eliminate the others by moving prices). Alternatively, if MaxConcurrentTrades > 1, it could attempt more than one concurrently if they involve different tokens/exchanges with minimal overlap. The output of this detection phase is one or more viable arbitrage opportunity objects that include: the token cycle (A, B, C), the exchanges for each leg (X, Y, Z), and the current prices on each leg.
The detection logic runs continuously as market data updates. TradeMonitor’s data collection will provide updated order books or quotes from each exchange at the configured interval. SolanaCrossDexTriangle may use a ScannerBase or similar mechanism (as hinted by TradeMonitor’s design) to iterate through all token pairs on all exchanges in SupportedTokens/Exchanges. This can be optimized by only scanning tokens that have at least two connections (since a triangle needs three tokens). The approach effectively maps all possible trading connections and then searches those connections for loops. To illustrate, imagine a simplified graph of three exchanges and four tokens: SOL, USDC, BONK, USDT. If Raydium has pools SOL/USDC and SOL/BONK, Orca has pools USDC/BONK and USDC/USDT, and Jupiter can swap BONK/USDT etc., the strategy will link these and possibly find a cycle like SOL -> USDC (Raydium), USDC -> BONK (Orca), BONK -> SOL (Jupiter) as a triangle. It will then use real-time rates on each to check if the SOL→SOL round-trip yields > 1 SOL. The graph-based detection ensures all such combinations are evaluated systematically rather than hard-coding specific token triplets
degencode.com
. (Note: By focusing only on triangles, we deliberately keep paths of length 3. Longer arbitrage loops (with more hops) are beyond the scope here due to increased complexity and risk. Three is the sweet spot balancing complexity and opportunity.)
Trade Simulation & Slippage-Adjusted Profit Estimation
Once a candidate arbitrage cycle is identified, the strategy simulates the trade to estimate the actual profit, accounting for slippage and fees. This simulation step is part of the Detailed Analysis phase before execution. The process is as follows for a given cycle (A → B → C → A):
Determine Trade Amount: The strategy needs to decide how much of the starting token A to use for the simulation (and potentially execution). It could use a nominal amount (e.g. 1 unit of A) to calculate a rate, or it could choose the maximum feasible amount. A common approach is to start with 1 unit of A to compute the conversion factor (product of rates). Then, for scaling, consider pool liquidity – the profit percentage often diminishes as trade size increases due to AMM slippage. The strategy might employ a fraction of MaxTradeAmount or run multiple simulations at different amounts. In practice, a reasonable choice is to target an amount that yields a good balance of profit and low slippage, or simply use MaxTradeAmount (but capped by pool size limits) for simulation if aiming to utilize full capacity.
Simulate Leg-by-Leg Conversion: Using the current order book or AMM formula for each leg:
Leg 1 (A → B on Exchange X): Query the exchange or use its pricing formula to estimate how much of B you’d get by swapping a given amount of A. For AMMs, the constant-product formula can be used: if the pool has reserves (R<sub>A</sub>, R<sub>B</sub>), swapping ΔA of token A yields ΔB of token B = R<sub>B</sub> * ΔA / (R<sub>A</sub> + ΔA) after fees. The exchange’s API or the TradeMonitor market data likely provides a function to get output given input. We also subtract the DEX’s swap fee (e.g. 0.25-0.30%) in the calculation. The result is the intermediate amount of token B.
Leg 2 (B → C on Exchange Y): Take the output of leg1 as input and simulate the swap on exchange Y similarly, yielding some amount of token C.
Leg 3 (C → A on Exchange Z): Simulate swapping the result of leg2 from C back to A on exchange Z.
During this simulation, the strategy applies the configured SlippageTolerance. This can be done in two ways:
Pessimistic Output: Assume that each leg’s price moves against us by the slippage tolerance. For example, if tolerance is 0.5%, multiply the output of each leg by (1 - 0.005) as a safety margin. Essentially, forecast slightly worse rates than current. This gives a conservative estimate of final A output.
Check Tolerance: Alternatively, compute the exact outputs with current data; then at the end verify that the final output of A exceeds the initial A by at least MinimumProfitThreshold and that even if each leg slipped 0.5%, it would still be profitable. If not, mark the opportunity as too sensitive to slippage.
Profit Calculation: Compare the final simulated amount of A versus the initial amount. For example, if starting with 100 A, and after A→B→C→A simulation we project ending with 102 A, that’s a 2% profit (before gas fees). The net profit is 2 A (which in value might be multiplied by token price if needed). We then ensure 2% >= MinimumProfitThreshold. Additionally, subtract any explicit costs:
Transaction fees: On Solana, transaction fees are usually small (a few hundred lamports ~ fractions of a penny), but if using a priority fee or if many transactions, it could add up. In high-throughput scenarios or if using Jito, fees/tips might be nontrivial. The strategy can factor a fixed cost per leg or per transaction into the profit calc.
Opportunity cost of time: If MaxExecutionTimeSeconds is large, perhaps skip if profit is so small that price could change in that time. (This is indirectly handled by requiring profit > threshold).
If any leg requires a bridge or unwrap (not likely here since all in Solana SPL tokens), that cost would be included too.
Output a TradingOpportunity: If after all these considerations the cycle yields, say, X% profit which is above the threshold, the strategy creates an opportunity object. This object contains:
The path: A → B → C → A and the specific exchanges for each hop.
The recommended trade amount (the strategy might use the full MaxTradeAmount or a smaller amount that it deems optimal).
The estimated profit (both in absolute terms of A and as a percentage).
The expected slippage or confidence score (perhaps how close to the threshold it is, or a rating of opportunity quality).
Any other info needed for execution (like the exact sequence of swap instructions, or the min acceptable outputs per leg per slippage tolerance).
During this simulation, the strategy must be wary of order book depth and pool liquidity. For example, a cycle might look highly profitable for 1 unit of A, but if the pool is shallow, trying to trade 1000 A could dramatically move the price and erase profit. A robust approach is to simulate using the actual intended trade size (or a range up to that size). In some implementations, one might even do a binary search on trade size to find the optimal amount that maximizes profit without going under threshold. However, to keep things simpler and fast, SolanaCrossDexTriangle might just pick a conservative size or the max and ensure profit still > 0. If the profit is higher with a smaller trade (due to slippage), it could choose that instead of max. The configuration’s MinTradeAmount also comes into play – if the simulation suggests the optimal amount is below MinTradeAmount, the strategy may discard the opportunity (not worth the effort). One more element considered is fees and rewards: Solana DEXs sometimes have LP fees (which we accounted for) and possibly referral rebates (not usually significant for strategy calculation). The strategy’s profit calc is net of all known fees. After simulation, any opportunity that doesn’t clearly exceed the profit threshold (with slippage buffer) is dropped. This prevents false positives from going to execution. Only high-confidence, positive expected value trades reach the next phase.
Execution Path: Sequential vs. Atomic Transactions
Executing a cross-DEX triangular arbitrage presents a choice: perform the three swaps one after the other (sequentially), or attempt to execute them all within a single atomic transaction. The SolanaCrossDexTriangle strategy is designed to handle both approaches and even use one as a fallback for the other, depending on practicality and risk.
Sequential Execution: In this mode, the strategy (via the bot and CommandChannel) will execute each leg as a separate trade in sequence. The typical flow:
Place order on Exchange X to swap Token A for Token B.
Upon success, immediately place order on Exchange Y to swap the obtained B for Token C.
Upon success of leg 2, place order on Exchange Z to swap C back to A.
The strategy ensures these steps occur back-to-back, and it will monitor the execution time so that if it exceeds MaxExecutionTimeSeconds overall, it may abort remaining legs. In TradeMonitor, each leg could be an individual TradeCommand issued to the CommandChannel, and the strategy/bot waits for confirmation of completion before issuing the next (or the commands are queued in order with dependencies). Pros: Sequential execution is simpler to implement and doesn’t require special multi-instruction transaction building. Each trade is a normal swap through the exchange API. TradeMonitor can rely on exchange confirmations at each step. It’s also easier to debug and log each leg’s result. Cons: There is execution risk – after leg 1, the market might move such that leg 2 or 3 is no longer profitable or fails. For example, by the time you reach the third swap, the final rate might have dropped, turning the cycle into a loss. Additionally, doing three separate transactions means paying three times the transaction fees (though on Solana that’s minor) and spending more time (market exposure between legs). If any leg fails (say due to slippage or network issue), you could be stuck holding token B or C which might require unwinding at a loss. The strategy’s risk management must handle this: for instance, if leg2 fails, perhaps issue a reverse trade of leg1 to revert to A, accepting a small loss, or some pre-defined stop-loss action in such cases.
Atomic Execution: Solana’s blockchain allows composing multiple swaps into one transaction (thanks to the ability to call multiple programs in one atomic transaction). An atomic arbitrage transaction would bundle the instructions for Raydium, Orca, and Jupiter swaps all together. When submitted:
All three legs execute within the same block, as one unit. If any part of the transaction fails (e.g., the final amount is below a specified threshold), the entire transaction is rolled back
docs.solanamevbot.com
. This means the arbitrage either fully succeeds or has no effect (except incurring a transaction fee). The trader takes no partial fill or inventory risk – it's all-or-nothing.
On Solana, this is powerful because you can eliminate leg-by-leg risk. The strategy can construct such a transaction using either a custom on-chain program/transaction builder or by leveraging an aggregator like Jupiter’s API in a special way. Jupiter, for instance, could potentially route a multi-hop swap that effectively covers the whole cycle, but generally Jupiter finds best A→A routes as no-ops. Instead, advanced bots craft their own transactions calling each DEX program in sequence

. This requires low-level knowledge of the DEX programs (Raydium, Orca, etc.) – essentially, one needs to include instructions like RaydiumSwapInstruction(A->B), OrcaSwapInstruction(B->C), Raydium/Serum/JupiterSwapInstruction(C->A) targeting the respective program IDs, and ensure the user’s token accounts are prepared for each swap. All these instructions share the same signed transaction. TradeMonitor could support this by allowing the strategy to create a composite Command or a custom CrossDexSwapCommand that encodes multiple swaps. Pros: No execution risk due to other bots jumping in mid-cycle or price movements during the cycle – if the profit condition fails by the time of execution, the transaction will simply not execute (it will revert). This is the safest financially, as the trader won’t end up holding unwanted tokens. It’s also faster in wall-clock time (one transaction vs three), which in a competitive environment can be the difference between capturing the arb or missing it. Cons: Building atomic transactions is complex. You must gather all needed accounts and instructions in advance. The combined transaction might hit size or compute limits if not optimized (though three swaps is usually fine on Solana’s limits). Also, atomicity means if it fails, you gain nothing (whereas sequentially, you might sometimes salvage something if two legs succeeded and only third failed, albeit that usually still results in some mismatched token in hand). Another challenge is finding the opportunity fast enough – many Solana MEV bots use atomic arbitrage with flash loans, and competition is high. This requires the strategy to be extremely low-latency in detection and submission

Fallback Handling: The strategy can employ a hybrid approach. For example:
It may attempt an atomic execution if possible (for instance, if the platform provides a way to bundle instructions easily). If the atomic transaction fails to land or is not supported (maybe due to an API limitation), the strategy could fall back to sequential execution. This could be configured: e.g., ExecutionMode: "AtomicPreferred" vs "Sequential".
Conversely, one might default to sequential but have a safeguard: if the sequential first leg succeeds but second fails, perhaps then try to quickly complete the cycle via Jupiter aggregator as a fallback to unwind. Or if atomic fails (i.e., transaction was not confirmed or ran out of compute), maybe revert to sequential if the window is still open (though likely if atomic failed due to slippage check, sequential would fail too).
The CommandChannel can be used cleverly here: For sequential mode, issue commands one by one. For atomic mode, issue one composite command. One could even implement a conditional command: "Try atomic; if not executed, do sequential legs".
In all cases, the strategy includes risk guards to handle failures. For example, TradeMonitor could have a built-in mechanism that if a multi-leg strategy only partially executed, it triggers a circuit breaker or at least an alert. The strategy might mark the opportunity as stale after a certain time or if one leg fails. Because the SolanaCrossDexTriangle is intended to complete cycles, any deviation (ending up with intermediate token) is considered an error state to resolve. Typically, the strategy/bot would immediately attempt to roll back the position: e.g., if leg1 executed and leg2 didn’t, now the bot holds token B; it might either try to swap B→A on the same exchange or another to minimize loss (essentially aborting the triangle) – possibly using Jupiter aggregator to convert B back to A at the best rate, which might incur a small loss but avoids holding B. These decisions can be pre-coded or configured as part of the risk management (some might choose to hold the token and wait if they believe prices will mean-revert, but that strays from arbitrage into speculation, so default is to unwind immediately). In summary, SolanaCrossDexTriangle will aim for atomic execution on Solana whenever feasible (because of the clear advantage of guaranteed outcome)

Example Arbitrage Scenario
To concretely demonstrate how SolanaCrossDexTriangle works, let’s walk through a hypothetical arbitrage opportunity step by step. This example uses three exchanges (Raydium, Orca, Jupiter) and three tokens (SOL, USDC, BONK) to illustrate a cross-DEX triangular loop: Scenario: Assume the following market conditions (deliberately exaggerated for clarity):
Raydium (Exchange X) SOL/USDC pool price is 1 SOL = 25 USDC. (SOL is slightly underpriced on Raydium.)
Orca (Exchange Y) USDC/BONK pool price is 1 USDC = 200 BONK. So 25 USDC would buy 5,000 BONK. (BONK is underpriced relative to USDC here perhaps.)
Jupiter (Exchange Z, aggregator) shows that 5,000 BONK can be swapped to 1.10 SOL (via some route, Jupiter finds perhaps a BONK->SOL path involving another DEX or Serum order book). This implies effectively 5,000 BONK = 1.10 SOL.
Now, consider a trader starting with 1 SOL in their wallet. The SolanaCrossDexTriangle strategy identifies a cycle: SOL -> USDC -> BONK -> SOL using (Raydium, Orca, Jupiter) respectively. Let’s examine the loop:
Leg 1 (SOL → USDC on Raydium): The trader swaps 1 SOL on Raydium and receives 25 USDC (ignoring minor pool fees for simplicity).
After Leg1: 0 SOL (spent), 25 USDC held.
Leg 2 (USDC → BONK on Orca): The 25 USDC is immediately used on Orca to buy BONK. At the rate of 200 BONK per USDC, the trader gets 25 * 200 = 5,000 BONK (again, minor fees aside).
After Leg2: 0 USDC, 5,000 BONK held.
Leg 3 (BONK → SOL on Jupiter): The 5,000 BONK is swapped via Jupiter aggregator back into SOL. Jupiter finds a route (maybe splitting across DEXs or order books) and yields 1.10 SOL for 5,000 BONK (which matches a price of ~4545 BONK per SOL, meaning BONK was relatively overvalued on that path).
After Leg3: 1.10 SOL, 0 BONK remaining.
After completing the cycle, the trader now has 1.10 SOL, having started with 1 SOL. This is a net profit of 0.10 SOL (10% gain) minus any transaction fees. This clearly exceeds any reasonable profit threshold. The strategy would have detected this because the product of conversion rates is:
Raydium: 1 SOL → 25 USDC.
Orca: 25 USDC → 5,000 BONK.
Jupiter: 5,000 BONK → 1.10 SOL.
The effective product rate is 1 SOL → 1.10 SOL (a 1.10x cycle). In practice, the strategy would calculate: 25 * (5000/25) * (1.10/5000) in terms of starting SOL, which indeed equals 1.10. Execution Path: The strategy could execute this atomically: one transaction calling Raydium’s and Orca’s swap instructions and Jupiter’s program, ensuring that the final SOL output is at least, say, 1.05 SOL (if we set a limit) so that if any leg slipped, the transaction would abort unless at least that amount is returned. If atomic execution is used, either the entire cycle happens and the wallet ends with ~1.10 SOL, or it fails and the wallet still has 1 SOL (no partial trades). If sequential execution was used, the bot would do the Raydium swap, then the Orca swap, then Jupiter. Given the strong profit, even if slight slippage occurred, the final result might be, for instance, 1.08 SOL – still profitable. The strategy’s slippage tolerance (say 0.5%) would allow some variation. If, however, between leg 2 and 3 another trader had just pumped BONK price down such that 5,000 BONK now only yields 0.99 SOL on Jupiter, the strategy’s slippage check would ideally abort the trade (or if sequential, the Jupiter leg might fail due to the minimum output condition specified). Hypothetical Outcome: In this example, the profit is realized in SOL terms. The bot would record a successful arbitrage of +0.10 SOL (which, if SOL is say $20, that’s $2 profit on this small trade). The cycle also benefited all exchanges’ liquidity in a small way (by balancing prices). In reality, such a large discrepancy (10%) is rare and would be quickly eliminated by arbitragers; our strategy aims to capture much smaller discrepancies, perhaps 0.2% to 1%, with higher frequency and volume. Tokens: USDC, USDT, SOL. Exchanges: Orca (USDC/USDT pool), Raydium (SOL/USDC pool), Jupiter (USDT→SOL route).
Suppose 1 SOL = 25 USDC on Raydium; Orca has 25 USDC = 26 USDT (meaning USDT is slightly cheaper, so 1 USDC buys 1.04 USDT due to imbalance); and Jupiter can swap 26 USDT to 1.02 SOL.
Starting with 1 SOL:
Raydium: 1 SOL → 25 USDC,
Orca: 25 USDC → 26 USDT,
Jupiter: 26 USDT → 1.02 SOL.
End up with ~1.02 SOL, profit 2%. This is a more plausible small arbitrage. The strategy would spot this A->B->C->A cycle (SOL->USDC on Raydium, USDC->USDT on Orca, USDT->SOL on Jupiter) and execute accordingly.
